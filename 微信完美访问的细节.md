# 微信完美访问的细节

> 适用项目：`/vol2/1000/docker/cam-control`
> 目标：保证 `http://c6666.com:18880` 在手机微信与 PC 浏览器都稳定可用。

## 1. 稳定入口（唯一口径）
- 对外统一入口：`http://c6666.com:18880`
- 历史稳定路径：`/cam` 或 `/cam/`
- 不要并行维护多套入口语义（`/`、`/cam`、`/webrtc` 随意混跳），否则容易出现“页面开了但视频不出”。

## 2. 必须同时成立的三件事
- 前端里 WebRTC 地址（`index.html`）
- Nginx 路由（`web/default.conf`）
- MediaMTX 实际端点（`:8889` 的 `reader.js` / `whep`）

三者必须一一对应；改其中一个，不改其他两个，微信里大概率会失败。

## 3. 微信最容易失败的点
- `POST/OPTIONS /cam/whep` 返回 404：WebRTC 协商链路断。
- 前端“是否 WebRTC 模式”判断与 URL 结构不一致：会走错分支，不拉流。
- 使用跨域/异端口资源：微信更容易报 `Load failed`。
- `/cam` 与 `/cam/` 不兼容：微信内跳转经常会触发这个坑。
- HTML/JS 缓存没清：看起来像“改了没生效”。
- 挂载文件权限不对：Nginx 读不到文件，出现 403/404。

## 4. 生产建议（保持当前可用状态）
- 优先保持当前已验证可用的路由与页面逻辑，不做结构性重写。
- 若必须改动，按“前端地址 + Nginx 路由 + MediaMTX端点”整套联改。
- 入口 HTML 建议 `no-store/no-cache`，避免微信吃旧缓存。
- `web/index.html`、`web/default.conf`、`web/reader.js` 建议权限至少 `644`。

## 5. 发布前自检（必须全部通过）
在 NAS 执行：
## 6. 最终发现微信并不是不稳定，域名c6666.com没有备案，国产软件对于非备案域名禁用，只要域名后面带端口号就友好，纯域名模式不予通过

```bash
# 入口与页面
curl -I http://c6666.com:18880
curl -I http://c6666.com:18880/cam/

# WebRTC 资源
curl -I http://c6666.com:18880/cam/reader.js
curl -i -X OPTIONS http://c6666.com:18880/cam/whep | head -n 20

# 容器状态
cd /vol2/1000/docker/cam-control
sudo docker compose ps
sudo docker logs --tail 120 cam-web
sudo docker logs --tail 120 mediamtx-cam
```

判定标准：
- 页面入口返回 `200`（或按设计返回可预期跳转）
- `/cam/reader.js` 返回 `200`
- `/cam/whep` 的 `OPTIONS` 不应为 `404`
- `mediamtx-cam` 日志能看到 WebRTC session（`created` / `peer connection established`）

## 6. 变更纪律（避免再次踩坑）
- 只改一处的 PR 一律拒绝：涉及 WebRTC 路由时，必须同时提交前端与 Nginx 对应改动。
- 改完必须附自检结果（上面的 6 条命令输出结论）。
- 出现“微信不出画”优先查 `/cam/whep`，不要先改 UI。

## 7. 回滚策略
- 优先回滚到最近已验证 commit（当前仓库 `git log` 中可用版本）。
- 回滚后立刻检查文件权限（`chmod 644 web/index.html web/default.conf web/reader.js`）。
- 然后重启：

```bash
cd /vol2/1000/docker/cam-control
sudo docker compose restart cam-web
```

---

维护原则一句话：
**这个项目的可用性来自“入口、路由、协商链路”一致性，任何单点改动都会破坏微信稳定性。**
